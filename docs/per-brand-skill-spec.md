# Per-Brand SKILL.md Specification

Each extracted brand produces a per-brand `SKILL.md` — a condensed, operational skill file that Claude auto-loads when the user mentions the brand's visual identity.

## What a per-brand SKILL.md is

The per-brand SKILL.md is a Claude Code skill file installed into a project's `.claude/skills/` directory. When Claude detects matching trigger phrases in the user's prompt, it loads the skill and applies the brand's design rules to any UI work in that session.

The SKILL.md is generated by `agents/skill-packager.md` during Phase C of extraction. It condenses the full DESIGN.md into an operational subset: inline token values, component rules, and voice guardrails — everything Claude needs to produce on-brand UI without consulting the full spec.

## Why negative triggers matter

Brand names collide with product names. "Stripe" is both a design system and a payment API. "Linear" is both a design system and an issue-tracking product. Without negative triggers, asking Claude to "integrate the Stripe API" would load the Stripe design skill and apply cobalt-and-white styling to your API integration code.

Negative triggers are explicit exclusions in the skill's `description` field and body. They tell Claude: "if the user is talking about the product's *functionality*, not its *visual identity*, do not activate this skill."

Example from the Nimbus seed brand:

```
Do NOT trigger for cloud infrastructure architecture, AWS, GCP, Azure,
Kubernetes, container orchestration, deployment pipelines, devops tooling,
IaC, Terraform, or any real cloud provider topic — Nimbus is a synthetic
design brand, not a cloud product.
```

## Schema

### Frontmatter

```yaml
---
name: brand-<slug>
description: >
  Apply <Brand>'s design system — <1-line summary of visual identity>.
  Use when "<trigger phrase 1>", "<trigger phrase 2>", ...
  Do NOT trigger for <negative triggers>.
version: <semver>
source_brand: <slug>
source_url: <url>
extracted_at: <date>
synthetic: <boolean>  # only for seed data
---
```

The `description` field is the most important. It must be densely packed with positive trigger phrases (at least 6) and negative triggers. Claude matches against this field to decide whether to auto-load the skill.

### Body sections

#### 1. Title and context

```markdown
# <Brand> — per-brand design skill

> Auto-installed by `/design-extractor:apply-design <slug>`.
```

#### 2. When to use this skill

Bulleted list of 8-12 positive trigger phrases. These are the prompts that should activate the skill.

#### 3. When NOT to use this skill

Bulleted list of negative triggers grouped by category (product features, competing products, homonyms). Ends with a summary rule.

#### 4. One-sentence summary

A single sentence capturing the brand's visual identity in enough detail to guide generation without reading further.

#### 5. Core tokens (inline)

The critical token values inlined directly in the skill body — not referenced from an external file. This ensures Claude has the values immediately upon skill load. Organized as:

- Colours (surface, text, border, accent, state)
- Typography (families, weights, scale)
- Spacing (base unit, scale, key measurements)
- Radii
- Shadows
- Motion (duration, easing)

#### 6. Component rules

Per-component numbered rules for the major UI elements (button, card, hero, nav, form). Each rule specifies exact token values. These are imperative ("Primary: cobalt `#4A6FF5` bg, white text, **pill** radius") not descriptive.

#### 7. Voice guardrails

Do/don't rules for copy. Specific CTA patterns. Words and phrases to avoid. Tagline (verbatim, if the brand has one).

#### 8. Install command

The `/design-extractor:apply-design <slug>` command that installed this skill.

#### 9. Source

Relative paths to the full DESIGN.md, design-tokens.json, design-tokens.css, replica, and metadata.

### Progressive disclosure

The skill body must stay under 500 lines (per Claude Code skill conventions). For brands with complex systems, additional detail goes into `skill/references/`:

- `references/tokens-full.json` — complete W3C DTCG token file
- `references/components.md` — extended component documentation
- `references/voice-corpus.md` — full voice analysis with examples

The skill body references these files but does not inline them. Claude loads them on demand when deeper detail is needed.

## Installing a brand skill

```
/design-extractor:apply <slug>
```

This runs `agents/librarian.md` which copies the skill to:

```
<project>/.claude/skills/brand-<slug>/SKILL.md
```

Claude auto-discovers skills in `.claude/skills/` and loads matching ones based on the `description` field.

## Testing trigger precision

Every per-brand SKILL.md should be validated with 10 test phrases:

**5 should-trigger phrases** — prompts that should activate the skill:
1. "design this like `<brand>`"
2. "`<brand>`-style landing page"
3. "apply the `<brand>` brand to this component"
4. "use `<brand>` tokens for this card"
5. "make this feel like `<brand>`"

**5 should-NOT-trigger phrases** — prompts that must not activate the skill:
1. "`<brand>` API integration"
2. "how does `<brand>` work" (product question)
3. "`<brand>` pricing" (product feature, not design)
4. A competing product name in the same space
5. A homonym or unrelated use of the brand name

This test matrix is part of the Phase 6 validation checklist in `blueprints/scaffolding-notes.md`.
